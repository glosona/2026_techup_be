# 0409(목) - CI/CD 배포(GitHub Actions & AWS ECR)

## 1. 배포 전략

1. 수동 배포: EC2에 직접 .jar 실행
   - 배포 기록이 남지 않는다.
2. CI/CD를 통한 자동 배포: GitHub Actions나 Jenkins로 코드 빌드하고 EC2에 명령 내리는 방식
   - 컨테이너 교체를 위해 기존 컨테이너를 죽이는 순간 `Downtime(서비스 중단 시간)`이 발생
3. 무중단 배포(Zero-Downtime Deployment): 서비스 중단 없이 새 버전 배포
   1. `Rolling`: 서버를 한 대씩 순차적으로 교체하는 방식. LB에서 제외/복귀하며 다른 서버들이 트래픽을 처리하게 한다.
   2. `Blue-Green`: 구버전(Blue)와 동일한 수의 신버전(Green) 서버를 띄운 후, LB 설정을 변경해 트래픽을 한번에 전환하는 방식
      - 즉각적인 롤백 가능
      - 일시적으로 시스템 자원이 2배 필요
   3. `Canary`: 일부 사용자에게만 새 버전 먼저 노출하는 방식
      - 안정성과 오류 여부를 안전하게 검증
      - 트래픽 분산 제어 로직이 복잡해질 수 있다.

<br>

## 2. 규모별 배포 방식

| 레벨                          | 인프라                  | 배포 방식                                             | 특징                                                                                |
|-----------------------------|----------------------|---------------------------------------------------|-----------------------------------------------------------------------------------|
| Level 1<br/>(서비스 초기)        | EC2+RDS              | GitHub Actions<br/>→ SSH(또는 SSM)                  | - 가장 저렴<br/>- Downtime 발생<br/>- OS 업데이트나 보안 패치를 직접 관리해야 함                         |
| Level 2<br/>(사용자 증가)        | ALB+EC2(여러 대)+ASG    | GitHub Actions<br/>→ AWS CodeDeploy               | - 무중단 배포<br/>- 자동 롤백<br/>- ASG로 트래픽 몰리면 서버 대수가 자동으로 늘어남                           |
| Level 3<br/>(운영 효율화)        | AWS ECS(Fargate)     | GitHub Actions<br/>→ ECR<br/>→ ECS Service Update | - Serverless 컨테이너: EC2 서버를 직접 관리할 필요가 없음<br/>- 강력한 격리<br/>- 배포 속도가 빠르고 설정이 표준화된다. |
| Level 4<br/>(대규모 서비스 및 MSA) | AWS EKS (Kubernetes) | GitOps(ArgoCD)                                    | - 가장 강력항 확장성과 유연성 제공<br/>- 글로벌 서비스나 대규모 트래픽 처리에 최적                                |

## 3. CI/CD

- `CI(Continuous Integration, 지속적 통합)`: 공유 저장소에 통합할 때 자동으로 빌드 및 테스트 수행(오류 조기에 발견)
- `CD(Continuous Delvery/Deployment, 지속적 배포/제공)`: CI를 통과한 코드를 프로덕션 환경에 배포할 수 있도록 준비, .jar를 실제 서버에 자동으로 배포하는 것

<br>

## 4. CI/CD 배포(AWS ECR)

### a) 시나리오

```plaintext
1. git push
2. GitHub Action 실행
  - Gradle 테스트 및 빌드
  - AWS 인증 (OIDC)
  - ECR 로그인
  - Docker 빌드 및 푸시
  - EC2 배포 (AWS SSM send-command 실행)
3. AWS System Manger(SSM)
  - EC2 내부의 SSM Agent가 명령 수신
  - EC2가 ECR에서 이미지를 Pull하고 컨테이너 재실행
  - 기존 컨테이너 다운, 새 컨테이너 실행(Downtime 발생)
```

<br>

### b) AWS 인증 - OIDC(OpenID Connect)

OIDC: AccessKey, Secret Key 대신 믿을 만한 제3자가 보증해주는 방식

* 배포할 때마다 AWS가 약 1시간만 유효한 임시토큰을 발행한다. 배포가 끝나면 무효된다.
* 조건을 사용할 수 있다: 특정 레포의 main에서 온 요청만 허용 등등
* IAM Role 생성:
  * ID 제공업체: `token.actions.githubusercontent.com`
  * Audience: `sts.amazonaws.com`

### c) GitHub Actions

**`.github/workflows/ci-cd.yml`**
- Test, AWS 인증, ECR 로그인, Docker 빌드 및 푸시, EC2 배포 단계로 구성
- `Deploy to EC2 via SSM` 단계의 `docker stop → docker rm → docker run` 과정에서 Downtime 발생

<br>

**GitHub Actions IAM 역할 권한**
- `AmazonEC2ContainerRegistryPowerUser`: 빌드한 이미지를 ECR에 Push하는 권한
- 인라인 정책으로 GitHub Actions → SSM send-command 요청 허용

<br>

### d) AWS 구성요소

**SSM (AWS Systems Manager)**
- EC2 내부의 SSM Agent가 명령을 수신하고, ECR에서 이미지를 Pull해 컨테이너를 재실행
- EC2에 IAM 역할 할당 필요
    - `AmazonSSMManagedInstanceCore`: SSM이 EC2를 제어할 수 있는 권한
    - `AmazonEC2ContainerRegistryReadOnly`: ECR에서 이미지를 Pull할 수 있는 권한
- Private Subnet EC2는 인터넷 연결이 없으므로, SSM 통신을 위해 Public Subnet에 NAT G/W 설치 필요

<br>

**ECR (Elastic Container Registry)**
- Private Repository로 생성해 Docker 이미지를 관리
