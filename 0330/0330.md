# 0330(월) - Java 동시성 관리

---

## 1. 동시성 문제

> 여러 스레드가 동일한 공유 자원에 동시에 접근(읽기-쓰기)할 때 발생하는 **데이터 불일치 문제**

```java
T1 → 재고 조회 → stock = 1
T2 → 재고 조회 → stock = 2
T1 → 재고 차감 → stock = 0   // 성공
T2 → 재고 차감 → stock = -1  // ⚠️ 데이터 정합성이 깨짐
```

**해결 방법:**
1. **락으로 임계 영역 보호** — `synchronized`, `ReentrantLock` 등
2. **원자적 연산(CAS)** — `AtomicInteger` 등

> **임계 영역 (Critical Section)**: 한 번에 하나의 스레드만 접근해야 하는 코드 영역
>
> 판단 기준:
> 1. 변수/자원이 여러 스레드에서 공유되는가?
> 2. 그중 쓰기가 포함되는가?
> 3. 둘 다 해당하면 → 임계 영역으로 보호

---

## 2. JVM 동시성 발생 영역

Runtime Data Area에는 **스레드가 공유하는 영역**과 **스레드별 독립 영역**이 있다.

| 영역 | 스레드 간 공유 여부 | 동시성 문제 |
|---|---|---|
| **Heap** | 공유 (Thread Unsafe) | 발생 가능 → 임계 영역 보호 필수 |
| **Method Area** | 공유 (Thread Unsafe) | 발생 가능 → static 변수 등 주의 |
| **Stack** | 스레드별 독립 (Thread Safe) | 발생 안 함 |
| **PC Register** | 스레드별 독립 (Thread Safe) | 발생 안 함 |

**Heap 동시성 문제 발생 예시:**

```java
private int count = 0; // Heap 영역 — 공유 자원

public static void main(String[] args) {
    for (int i = 0; i < threadCount; i++) {
        Thread thread = new Thread(counter::increaseCount); // ⚠️ 동시성 문제 발생 지점
        thread.start();
        threads.add(thread);
    }
}
```

여러 스레드가 동시에 `count`에 쓰기를 시도하기 때문에 실제 스레드 수와 `count` 값이 불일치한다.

## 3. 동시성 관리 방법

### a) 애플리케이션 락 (JVM 락)

#### 1) synchronized (기본 락) ⚠️

> Java 객체에 내장된 **모니터(Monitor)** 를 사용하는 가장 기본적인 동기화 메커니즘

- 한 스레드만 접근 가능. 락을 획득할 때까지 `BLOCKED` 상태
- 지정 범위가 끝나면 락 자동 반납

```java
private synchronized void increase() { ... }
// 또는
synchronized (this) { count++; }
```

> **모니터**: 상호 배제(Mutex)와 조건 대기(Condition wait)를 제공하는 동기화 구조체

**⚠️ 치명적인 단점:**

| 단점 | 내용 |
|---|---|
| **무한 대기** | 타임아웃 없음, 인터럽트 불가 |
| **공정성 없음** | 락 반환 시 대기 스레드 중 랜덤 선택 → 특정 스레드가 계속 대기하는 **기아 상태(Starvation)** 발생 가능 |

---

#### 2) ReentrantLock (고급 락) ✅

> **CAS(Compare-And-Swap)** 기반. 개발자가 락 획득/반납을 직접 제어

- **원자적 동작**: CPU가 읽기·쓰기를 하나의 연산으로 처리 → 각 스레드가 최신 값을 보장해서 읽음
- 하드웨어 지원 필요 (구형 Intel 일부 제외)
- `new ReentrantLock(true)`로 락을 FIFO로 부여하는 **공정모드** 설정 가능(synchronized의 기아 상태 해소)

```java
class Counter {
    private int count = 0;
    private final ReentrantLock lock = new ReentrantLock();

    public void increase() {
        lock.lock();  // 락 획득
        try {
            count++;
        } finally {
            lock.unlock();  // 락 반납 (finally에서 반드시 해제)
        }
    }
}
```

`tryLock()`으로 타임아웃 제어:

```java
while (true) {
    try {
        // 3초 동안 락 획득 시도 → 실패하면 while 루프로 재시도
        if (this.lock.tryLock(3, TimeUnit.SECONDS)) {
            try {
                log.info("락 획득");
                break;
            } finally {
                this.lock.unlock();  // 락 반납
            }
        }
    } catch (InterruptedException e) { ... }
}
```

---

**synchronized vs ReentrantLock 비교:**

| | synchronized | ReentrantLock |
|---|---|---|
| 타임아웃 | ❌ | ✅ (`tryLock`) |
| 인터럽트 | ❌ | ✅ |
| 공정성 설정 | ❌ | ✅ |
| 속도 | 느림 | 빠름 (CAS 기반) |
| 락 제어 | 자동 | 수동 (`lock/unlock`) |

### b) JPA Lock (DB 락)



#### 1) 낙관적 락 (Optimistic Lock)

> **애플리케이션 레벨**: `@Version` 사용
> "충돌이 거의 없을 것이다"는 낙관적 전제. DB에 실제 락을 걸지 않고 **버전 번호**로 충돌 감지.

```java
T1: version=1 → UPDATE 성공 → version=2로 자동 증가
T2: version=1 → UPDATE 실패 (0 rows 반환) → 예외 발생
```

```java
-- 실제 실행되는 쿼리
UPDATE ... WHERE id=1 AND version=1
```

Entity에 `@Version` 어노테이션으로 활성화:

```java
@Entity
public class Product {
    ...
    @Version  // 낙관적 락
    private Long version;
}
```

**⚠️ 단점:**

- DB 실제 락이 아니기 때문에 충돌 시 예외 → 롤백 → 재시도 반복 → **충돌이 잦을수록 비용 급증**
- e.g.) 재고 4개 차감 시도 → 충돌로 인해 3개만 성공할 수 있음
- 재시도가 필요하면 `@Retryable` 사용 (이때 `@EnableResilientMethods`를 Application 클래스에 추가)
    ```java
    @Retryable(
        includes = ObjectOptimisticLockingFailureException.class,
        maxRetries = 3
    )
    @Transactional
    public OrderResponseDto createOrderWithOptLock(OrderCreateDto dto) {
        ...
    }
    ```

---

#### 2) 비관적 락 (Pessimistic Lock)

> **DB 레벨**: `@Lock(LockModeType.PESSIMISTIC_WRITE)` 사용
> "충돌이 발생할 것이다"는 비관적 전제. DB에 실제 락(`FOR UPDATE` 등)을 걸어 다른 트랜잭션의 접근 차단.

- 락을 통해 접근 순서가 제어됨 → 데이터 정합성 강력 보장
- 충돌이 많은 환경에서 재시도 비용 없이 안정적 처리 가능

**⚠️ 단점:** 전체 처리량(throughput) 저하, 데드락 위험

---

**synchronized / ReentrantLock과 비관적 락의 차이:**

| | synchronized / ReentrantLock | 비관적 락 (`FOR UPDATE`) |
|---|---|---|
| 동작 범위 | 단일 서버, 같은 JVM 내 스레드 간 | **다중 서버에서도 동작** |
| 락 관리자 | JVM | **DB가 중앙 관리자 역할** |

---

#### 3) 낙관적 락 vs 비관적 락 선택 기준

| | 낙관적 락 | 비관적 락 |
|---|---|---|
| **충돌이 적을 때** | ✅ 유리: DB 락 없음 → 성능 이점, 데드락 위험 적음 | ❌ 불리: 매번 락 획득/해제 오버헤드 |
| **충돌이 많을 때** | ❌ 불리: 충돌 시 예외 → 롤백 → 재시도 반복으로 성능 저하 | ✅ 유리: 정합성 강력 보장, 재시도 비용 없음 |

---

#### 4) LockModeType 종류

| LockModeType | DB 락 | 설명 |
|---|---|---|
| `OPTIMISTIC` | 없음 | UPDATE 시 version 체크 |
| `PESSIMISTIC_READ` | `FOR SHARE` | 읽기 락 |
| `PESSIMISTIC_WRITE` | `FOR UPDATE` | 쓰기 락 |
| `PESSIMISTIC_FORCE_INCREMENT` | `FOR UPDATE` + version 증가 | DB 부하 증가, 쿼리 느려짐 |

---

### c) 분산 락 (Distributed Lock)

> 서버가 여러 대일 경우, **락을 관리하는 중앙 관리자**를 두는 것

- 다중 서버 환경에서는 JVM 레벨 락은 무의미
- **Redis 분산 락** 등을 통해 중앙에서 락 제어

```java
private final RedissonClient redissonClient;

public void order(Long productId, int quantity) throws InterruptedException {
    RLock lock = redissonClient.getLock("product:" + productId);  // 락 획득

    // 10초 대기, 락 보유 시간 3초
    boolean acquired = lock.tryLock(10, 3, TimeUnit.SECONDS);
    if (!acquired) throw new LockAcquisitionException("락 획득 실패");

    try {
        productService.decrease(productId, quantity);
    } finally {
        lock.unlock();  // 락 반납
    }
}
```

## 4. 동시성 관리 방법 전체 비교

| 방법 | 범위 | 적합한 상황 |
|---|---|---|
| `synchronized` | 단일 JVM | 간단한 동기화. 타임아웃 불필요 |
| `ReentrantLock` | 단일 JVM | 타임아웃·공정성 제어 필요 |
| 낙관적 락 | 단일 or 다중 서버 | 충돌이 드문 경우 |
| 비관적 락 | 단일 or 다중 서버 | 충돌이 잦고 정합성 최우선인 경우 |
| 분산 락 | 다중 서버 | 다중 서버 환경에서 락이 필요한 경우 |
